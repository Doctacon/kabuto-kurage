# Local Infrastructure as Code

This project uses local Infrastructure as Code to make the development runtime reproducible without pretending to operate production cloud infrastructure.

The IaC lives in:

```text
iac/local/
```

## What Terraform Manages

Terraform uses the open-source `hashicorp/local` provider only. It prepares ignored local runtime files under `.local/`:

| Resource | Purpose |
| --- | --- |
| `.local/dagster/dagster.yaml` | Creates a stable Dagster home config and disables Dagster telemetry by default. |
| `.local/runtime/kabuto.env` | Creates a non-secret env file for local commands and Docker Compose. |
| `.local/data/README.md` | Creates the ignored local data root where Delta tables are written. |

Terraform does **not** provision cloud services, Kubernetes clusters, networks, managed databases, secrets, or production deployment targets.

This is intentionally modest: the portfolio goal is to demonstrate Terraform concepts such as declared resources, variables, outputs, generated config, state, and repeatable local setup while staying honest about the project being local-first.

## What Docker Compose Manages

Docker Compose is optional and runs the Dagster UI service locally from the checked-out repository:

```text
iac/local/docker-compose.yml
```

Docker Compose is a better fit than Terraform for starting/stopping a local development process. Terraform prepares files; Compose runs the service.

Normal tests do not require Docker.

## Secrets

No token values are committed or generated by Terraform.

- `GITHUB_TOKEN` or `GH_TOKEN` should be exported in your shell or stored in ignored `.env` if you choose to use one.
- `.local/`, `.env`, `config/tenants.local.yaml`, Terraform state, and `.terraform/` provider directories are ignored by git.
- The generated `.local/runtime/kabuto.env` contains paths and safe defaults only.

## Prepare Local Runtime Files

From the repository root:

```bash
terraform -chdir=iac/local init
terraform -chdir=iac/local apply
```

Optional variable overrides:

```bash
terraform -chdir=iac/local apply \
  -var='tenant_config_path=config/tenants.local.yaml' \
  -var='github_max_repositories=1' \
  -var='dagster_port=3000'
```

Inspect generated outputs:

```bash
terraform -chdir=iac/local output
```

Use generated environment values in a shell:

```bash
set -a
source .local/runtime/kabuto.env
set +a
export GITHUB_TOKEN=...
uv run dagster dev -h 0.0.0.0 -p 3000 -m kabuto_kurage.definitions
```

Open Dagster at http://localhost:3000.

## Optional Docker Compose Dagster UI

After Terraform has generated `.local/runtime/kabuto.env`:

```bash
export GITHUB_TOKEN=... # optional until you materialize live GitHub bronze assets
docker-compose --env-file .local/runtime/kabuto.env -f iac/local/docker-compose.yml up dagster
```

Open Dagster at http://localhost:3000.

If your environment has the newer Docker Compose plugin instead of the standalone command, use `docker compose` with the same arguments.

Stop the service with `Ctrl-C`, or in another shell:

```bash
docker-compose --env-file .local/runtime/kabuto.env -f iac/local/docker-compose.yml down
```

## Validation

Static validation that does not require Docker daemon:

```bash
terraform -chdir=iac/local fmt -check
terraform -chdir=iac/local init -backend=false
terraform -chdir=iac/local validate
docker-compose --env-file .local/runtime/kabuto.env -f iac/local/docker-compose.yml config --quiet
uv run pytest
uv run ruff check .
uv run mypy src
```

`terraform apply` writes ignored local files and Terraform state. Do not commit generated state or local runtime files.

## Boundary Summary

- Terraform: local generated config and directory/file preparation.
- Docker Compose: optional local Dagster process runner.
- Python/Dagster: actual ingestion, transformations, metrics, and orchestration logic.
- Future production deployment: explicitly not chosen in this milestone.
